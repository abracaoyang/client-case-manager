# 客戶案件管理系統：中/高度開發可行性評估報告書

本報告書旨在評估「客戶案件管理系統」後續進行「中度開發（新增次步驟標籤）」與「高度開發（增員案件管理、月曆行事曆畫面、外部日曆聯動）」的可行性、影響範圍、技術架構以及開發注意事項，為後續接手開發者提供清晰的指引。

---

## 一、 中度開發：新增次步驟標籤

### 1. 業務場景
在現有的五大進度大階段（SA、OA、PC、C、S）中，因應業務流程調整，需要新增或移除某些細項步驟按鈕。

### 2. 技術可行性：**極高 (Easy)**
本系統採用彈性的前端資料對應與後端無痛同步設計，新增次步驟對系統原有架構無任何不良影響。

### 3. 接手開發步驟說明
若要以 `OA 階段` 新增一個「新步驟」為例：
1. **Google Sheet 欄位新增**：
   * 在試算表 `Sheet1` 中，新增兩個欄位：`"ＯＡ新步驟狀態"` 與 `"ＯＡ新步驟日期"`。
2. **後端 (code.gs) 對齊**：
   * 打開 `code.gs`，在頂端 `getOrCreateSheet()` 的 `targetHeaders` 陣列（約 Line 11 起）中，加入 `"ＯＡ新步驟狀態"` 與 `"ＯＡ新步驟日期"`，確保表頭順序與欄位完整對齊。
3. **前端 (index.html) 解析與對齊**：
   * **資料讀取（Deserialization）**：在 `saDetails` 或 `oaDetails`（約 Line 3320 起）中，加入新欄位的資料綁定：
     ```javascript
     newStepState: row["ＯＡ新步驟狀態"] || 'dim',
     newStepDate: row["ＯＡ新步驟日期"] || ''
     ```
   * **資料上傳（Serialization）**：在 `serialize` 函數中（約 Line 3394 起），將新狀態對應回試算表欄位名稱：
     ```javascript
     "ＯＡ新步驟狀態": oa.newStepState || 'dim',
     "ＯＡ新步驟日期": oa.newStepDate || ''
     ```
4. **主畫面 UI 按鈕渲染**：
   * 在 `renderCases()` 渲染 OA 階段按鈕 the 區塊中，比照其他次步驟新增一個按鈕節點（`<span>` 或 `<button>`），並綁定單雙擊事件。
5. **抽屜 (Drawer) 內部渲染**：
   * 在 `renderOADrawer()` 中加入對應欄位的輸入框（Input/DatePicker/Select），使使用者能在抽屜內編輯該步驟資料。

## 二、 高度開發：增員案件管理、月曆行事曆與外部日曆聯動

### 1. 業務場景
* 獨立一套「增員案件流程看板」，與現有客戶看板進行分頁切換。
* 新增全螢幕「月曆行程看板」，整合所有面談、約訪時段。
* 雙向或單向同步 Google Calendar，在網頁改期或約定時間時自動更新手機的日曆。

### 2. 技術可行性：**高 (Moderate - Hard)**
* **資料分流可行性**：極佳。可透過在 Google Sheet 中新增名為 `Recruits` 的分頁工作表，專門存放增員資料，前端只需透過參數向後端拉取不同分頁資料。
* **月曆看板可行性**：佳。建議在前端引入輕量級無相依性（Vanilla JS）的日曆套件，例如 `FullCalendar.js` 或是自研簡易 Grid 月曆。
* **Google Calendar 聯動可行性**：極佳。**這是 Google Apps Script 的原生優勢**。後端 `code.gs` 可以直接呼叫 Google 原生 `CalendarApp` 類別，在背景進行完全安全且自動化的日曆建立與改期同步。

### 3. 接手開發重點與技術提醒

#### ⚠️ 檔案結構模組化拆分（開發前必做）
* **現狀問題**：目前的 `index.html` 是一個超過 8700 行的超大型單一 HTML 檔案，裡面包含了極為龐大的 CSS、HTML 結構、待辦事項邏輯與五個階段抽屜的渲染邏輯。
* **風險**：如果在此結構下直接堆疊「增員看板」、「月曆」與「Google Calendar 聯動」，檔案將輕易突破 15,000 行。這會極大增加 AI 在上下文理解上的負擔，且非常容易在修改 A 功能時造成 B 功能無意間損壞。
* **建議接手作法**：
  * **在動工前，務必先將專案模組化拆分**：
    1. 將 CSS 獨立出 `styles.css`。
    2. 將通用 JS 函數、待辦模組、設定模組、以及各個大階段的抽屜渲染（SADrawer, OADrawer...）分拆成獨立的 `.js` 檔案。
    3. 在 `index.html` 中以 `<link>` 與 `<script>` 引入。
    4. 這樣做能確保代碼結構清晰、健壯，易於維護且防範 Bug。

#### ⚠️ Google 日曆權限授權
* **機制**：當 `code.gs` 第一次使用 `CalendarApp` 時，Google 系統會拋出權限驗證提示。擁有者（`abracaoyang@gmail.com`）必須在瀏覽器上點選「進階設定 ➔ 仍要前往 ➔ 允許授權」。

## 三、 開發最高指導原則 [ADHD 成人業務專屬]

不論是中度還是高度開發，任何接手開發者均須嚴格遵守以下系統設計指導：
1. **空間減法原則 (UX)**：不常駐顯示進階功能。非核心欄位一律預設隱藏，僅在點擊、雙擊或特定狀態觸發時滑出，拒絕資訊稠密。
2. **網格化對齊 (UI)**：所有輸入框、按鈕的寬高度必須網格化對齊，不可出現高低落差或參差不齊。
3. **微互動多巴胺回饋**：成功操作、階段推移時，必須給予明確的視覺反饋（如 Confetti 紙花動畫、色彩明顯跳轉、呼吸發光邊框）。
4. **雙擊防呆機制**：涉及進度推移、狀態切換的高風險操作，一律實作雙擊（onDoubleClick/ondblclick）觸發，防止誤觸。
5. **部署防護提醒**：任何涉及 `code.gs` 的變更，在交付時必須主動提醒使用者重新在 Apps Script 中部署為新版本，以防資料同步斷開。
